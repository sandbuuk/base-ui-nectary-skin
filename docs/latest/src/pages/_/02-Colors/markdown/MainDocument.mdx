# Colors

Like other values in the design system, we store colors as [Design Tokens](?path=/components/_/tokens). This allows us to maintain consistency and increases the flexibility and scalability of the system. All tokens are categorized into three main groups:

- **Reference** - don't have any specific meaning and used only as a reference for System and Component colors.
- **System** - contains colors following specific purposes, and shared between different components
- **Component** - they are component specific colors

Below you will find a detailed description of each of these categories.

## Reference colors

*Starts with `ref`/color/honey/400*

The Reference colors are aligned with Brand but don't have any specific meaning. Instead, they are used only as a reference for System and Component colors. As all the other styles are inherited from this base palette, it help us to maintain color consistency across the system and with brand.

Avoid using these colors directly.Instead, inherit from them.

## System colors

*Starts with `sys`/color/feedback/success/default*

This palette is designed to have colors following specific purposes. Consider this it as the **initial palette to go with.** They are the swatches you need first to inherit values for your local component styles. For example, if you need to emphasise that something is going as planned, consider using the `sys/color/success/default` as a reference value for your success icon style.

## Component colors

There is also a third set of colors designed for specific components. They are **not supposed to be used directly,** excepting the cases when you literally need to follow styling of a certain component. Otherwise, you may encounter issues if you use these styles improperly.

For example when you create a Super-Custom-Loading-Button and want it to follow the regular button style, you can use `comp/button...` styles as reference.
