# Examples

A sheet component is a slide-in drawer that allows users to view and interact with content in a panel that slides from the edge of the screen. It is typically used to display additional information, navigation options, or contextual actions without leaving the current page.

The sheet component consists of a container that slides in from one of the four edges (left, right, top, or bottom) and overlays or pushes the main content. It includes a title, scrollable content area, and optional action buttons. The sheet can be dismissed by the user by clicking a close button, clicking outside the sheet (for modal overlays), or pressing the ESC key.

Sheets are ideal for displaying detailed information, forms, settings panels, or navigation menus in a way that maintains context with the main application. They can be configured with different placement options and overlay modes to suit various use cases.

## Push Overlay

The sheet can be configured with `overlay="push"` mode, which pushes the main content aside instead of overlaying it. This example demonstrates how to listen to animation events and apply transitions to your page layout.

There are two ways to animate your content when using the push overlay mode:
1. **Using Animation Events**: The sheet emits `animation-start` event when the push animation starts. You can listen to these event to trigger animations on your content. Check the example below for implementation details.
2. **Using CSS Variables**: The sheet exports its dimensions via CSS variables (`--sinch-sheet-export-current-width` and `--sinch-sheet-export-current-height`) that can be used to animate your content. Check the Tokens section for more details.

## Animation
`animation-end` event is useful to know when the sheet has finished its opening or closing animation. This can be particularly helpful for triggering actions that should only occur after the sheet is fully visible or hidden, such as navigating to a different page, loading content, or updating the UI.

## Accessibility

The sheet component requires an `aria-label` attribute for screen reader accessibility. This label should describe the purpose of the sheet to users who cannot see it.

```tsx
<sinch-sheet
  open={isOpen}
  caption="Settings"
  aria-label="Settings panel"
  on-close={() => setIsOpen(false)}
>
  {/* content */}
</sinch-sheet>
```

## Note on sheet cancel events

Similar to the Dialog component, on Chrome-based browsers, if the user presses the `ESC` key twice, the second event that is produced isn't cancelable, meaning we can't prevent the default behavior of the browser hiding the sheet.

If you have an on-close callback looking like this:

```js
on-close={(e) => {
  if (e.detail === "escape") {
    // do nothing
    return
  }
}}
```

Consider adding this before:

```js
on-close={(e) => {
  if (!e.cancelable) {
    return setSheetOpen(false)
  }
  if (e.detail === "escape") {
    return
  }
}}
```
