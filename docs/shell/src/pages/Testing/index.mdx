# Testing

# Introduction

## What should you test

Your job is to test **your** application **NOT** the Nectary components (That's the responsibility of the Nectary team!).
However, in order to test your application, you would need to
interact with the Nectary components, but only through their interface.
The implementation details of the components are none of your concern.
You should only care about the public interface, [Read more here](https://www.artima.com/articles/design-principles-from-design-patterns).

## What is the purpose of testing

You are testing for your users.
Testing can also be a good thing for the development process itself; see [TDD](https://en.wikipedia.org/wiki/Test-driven_development).
However, we are not testing for code coverage.
Instead, testing should be similar to how your users interact with your application.
From a product quality standpoint, the best kind of tests are performed by real users in a real browser.
However, that would be impractical, so as software developers, we try to automate things.

## Our Recommendation: Playwright

We recommend that you test your application with Playwright. Playwright is used to test the Nectary components, so maintainability comes for free, meaning we can guarantee that it will work well.
Additionally, Playwright is more similar to how real users interact in a real browser compared to many other frameworks (like Testing Library and Jest with JSDOM), making it more reliable. Read more [here](https://playwright.dev/docs/why-playwright).

# Guides

## Main recommendation: Playwright for Testing Components

For component testing or unit/integration testing we recommend using Playwright CT

### Setup

For setting things up, it should be enough to follow the guide on the Playwright [website](https://playwright.dev/docs/test-components).
However, you will need to add the following snippet to the ````<rootDir>/playwright/index.tsx```` file

````tsx
import {setNectaryRegistry} from "@sinch-engage/nectary/utils/element";
import {setAssetsRegistry} from "@sinch-engage/nectary-assets/utils";
setAssetsRegistry(window.customElements)
setNectaryRegistry(window.customElements)
window.global ||= window;
````

### Example Tests

Below, we show a simple example component: a "sinch-button" connected to a "sinch-text" via a state variable called "count". The component you're building will probably have a more sensible use case.

<br />

`ButtonCounter.test.tsx`
````tsx
import { expect, test } from '@playwright/experimental-ct-react'
import { ButtonCounter } from './ButtonCounter'

test('Click on the button two times', async ({ mount }) => {
  const component = await mount(<ButtonCounter/>)
  await component.locator('[aria-label=Click]').click()
  /**The second click is to show that it's possible to select the Nectary
  components directly, but some people might regard this as an implementation detail. **/
  await component.locator('sinch-button').click()
  await expect(component).toContainText('counter is at current value 2')
})
````

## Testing Library+Jest

It's possible to test your code with Testing Library+Jest.

### Setup

Besides the steps we outlined in the [Intro](http://localhost:5000/?version=latest&path=/intro#testing).
Its should be enough to follow the steps on the Testing Library documentation [site](https://testing-library.com/).
Do not to forget to set the NectaryRegistry and AssetsRegistry like below in your tests.

````tsx
import { setNectaryRegistry } from '@sinch-engage/nectary/utils'
import { setAssetsRegistry } from '@sinch-engage/nectary-assets/utils'

setNectaryRegistry(window.customElements)
setAssetsRegistry(window.customElements)
````

### Example Tests

Below we show a simple example component: A sinch-button connected to a sinch-text via a state variable called count.
The component you're will build probably is a more sensible use-case.

`ButtonCounter.tsx`
<Code src={ButtonCounterExample.toString()} />

<br />

`ButtonCounter.test.tsx`


````tsx
import React from 'react';
import { setNectaryRegistry } from '@sinch-engage/nectary/utils'
import { setAssetsRegistry } from '@sinch-engage/nectary-assets/utils'
import { fireEvent, render, screen } from '@testing-library/react'
import { ButtonCounter } from './ButtonCounter'

setNectaryRegistry(window.customElements)
setAssetsRegistry(window.customElements)

test('Click Button, read resulting value from text field', async () => {
    render(<ButtonCounter/>)
    const button = screen.getByLabelText('Click')
    fireEvent(button, new CustomEvent('-click'));
    fireEvent(button, new CustomEvent('-click'));
    fireEvent(button, new CustomEvent('-click'));
    screen.getByText('counter is at current value 3')
})
````

## Cypress E2E

It's possible to test your code with Cypress E2E.
Here you can, if you want to, add [cypress-testing-library](https://testing-library.com/docs/cypress-testing-library/intro/), to benefit from its queries.

The component you're building probably has a more sensible use-case.

`ButtonCounter.cy.tsx`

````tsx
it("Button",  () => {
    cy.visit("/button");
    cy.findByLabelText('Click').then(async (element) => {
        element[0].dispatchEvent(new CustomEvent('-click'));
        cy.findByText('counter is at current value 1');
    });
});
````
