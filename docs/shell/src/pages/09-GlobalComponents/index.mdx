# Global Components

> This page is only relevant for Micro-Frontends

With v5.0 of `@nectary/components` and v3.0 of `@nectary/assets` comes official support for using globally registered Nectary elements in Micro-Frontends. This essentially means that MFEs passing `window.customElements` to `setNectaryRegistry()` will resolve components from a single global source (CDN) instead of using the local components bundled in the MFE.
> This assumes that the Host application has enabled Global Components.

## Component Registration
See the diagrams below to understand the difference of how custom elements were registered previously in the document vs after Global Components were enabled:

<img className="image" width="640" src={beforeImage} alt="Before"/>

In this hypothetical example, the red arrow highlights a potential conflict:

> MfeFoo and MfeBar has installed v3 of @nectary/components.

> MfeBaz has installed v4 of @nectary/components.

MfeBaz expects new functionality in sinch-button introduced in v4, but instead receives version 3 because MfeFoo was the first MFE to register sinch-button in the custom elements registry.

Global Components solves this by ensuring that every MFE resolves the components from a source where the version is stable:

<img className="image" width="640" src={afterImage} alt="After"/>

Here, the first time `sinch-button` is requested, it loads the latest version directly from the CDN. This ensures that every MFE always gets the same or a newer version of the component, preventing version conflicts. To support this, Nectary maintains forwards compatibility and avoids introducing breaking changes in future releases.

Because components are loaded on demand, you may notice a brief flicker the first time a component is used. To prevent this, the Host application can preload all components so they are ready and available as soon as an MFE loads.

## Optimized Imports

Because components are now loaded from an external source, you no longer need to include their local definitions in your bundle. To optimize bundle size, use the `/global` import path which excludes the local component definition:

```js
import '@nectary/components/button/global' // ✅ Optimized bundle size
```
```js
import '@nectary/components/button'     // ⚠️ Includes redundant local constructor code
```
